# 操作系统实验二报告参考答案

## 一、实验目的

本实验通过实现一个工业级高并发线程池，加深对线程复用、任务队列、互斥锁、条件变量、虚假唤醒、惊群效应以及优雅退出等概念的理解。

## 二、核心代码实现思路

### 1. Worker 线程逻辑

工作线程在无限循环中重复以下过程：

1. 先加锁；
2. 当任务队列为空且线程池未关闭时，调用 `pthread_cond_wait` 进入休眠；
3. 被唤醒后再次检查条件；
4. 若线程池已关闭且队列为空，则退出线程；
5. 若队列中有任务，则从队头取出任务；
6. 立即解锁；
7. 在锁外执行任务并释放任务节点。

### 2. Submit 逻辑

任务提交函数的主要流程：

1. 检查参数是否合法；
2. 在堆上创建任务节点；
3. 加锁后再次检查 `shutdown`；
4. 若线程池已关闭，则拒绝提交并释放任务节点；
5. 将任务插入队尾；
6. 调用 `pthread_cond_signal` 唤醒一个工作线程；
7. 解锁并返回。

### 3. Destroy 逻辑

线程池销毁函数的主要流程：

1. 检查线程池指针；
2. 加锁并设置 `shutdown = 1`；
3. 调用 `pthread_cond_broadcast` 唤醒全部等待线程；
4. 解锁；
5. 对所有工作线程执行 `pthread_join`；
6. 销毁互斥锁和条件变量；
7. 释放线程数组和线程池内存。

## 三、思考题答案

### 1. 为什么 `pthread_cond_wait` 必须放在 `while` 循环中，而不能只用 `if`？

因为条件变量存在**虚假唤醒**问题，也就是线程可能在没有真正收到任务通知时被唤醒。另外，即使真的被唤醒，也可能是别的线程已经先一步拿走了任务。如果只用 `if`，线程被唤醒后会直接往下执行，可能在队列为空时错误取任务。使用 `while` 可以让线程每次被唤醒后重新检查“队列是否为空且线程池是否关闭”，从而保证逻辑正确。

### 2. 为什么取出任务后要先释放锁，再执行 `task->function(task->argument)`？

因为任务函数通常是耗时操作，如果一直持有锁去执行任务，会导致其他线程无法访问共享任务队列，整个线程池会退化成串行执行，失去并发意义。先解锁再执行任务，可以让其他线程继续提交任务或取走任务，提高并发性能。

### 3. 为什么提交任务时需要在持锁状态下检查 `shutdown` 标志？

因为存在竞争条件：线程刚创建好任务节点，但还没加锁时，另一个线程可能已经调用了 `thread_pool_destroy` 并把线程池关闭了。如果不在持锁状态下再次检查 `shutdown`，就可能把任务错误地插入已经关闭的线程池。这样会造成逻辑错误，甚至资源管理混乱。

### 4. 为什么提交一个任务后使用 `pthread_cond_signal`，而不是默认使用 `pthread_cond_broadcast`？

因为一次通常只新增一个任务，只需要唤醒一个工作线程去处理即可。若使用 `broadcast`，会把所有等待线程都唤醒，但最终往往只有一个线程能成功拿到任务，其余线程又会重新休眠。这会造成大量无效唤醒、锁竞争和上下文切换，也就是“惊群效应”。因此提交单个任务时使用 `signal` 更高效。

### 5. 为什么关闭线程池时需要使用 `pthread_cond_broadcast` 唤醒所有工作线程？

因为线程池关闭时，可能有多个工作线程都阻塞在条件变量上。如果只唤醒一个线程，其余线程可能会一直睡眠，导致主线程在 `pthread_join` 时永远等不到它们退出。使用 `broadcast` 可以确保所有等待线程都被唤醒，让它们重新检查 `shutdown` 条件并安全退出。

### 6. 为什么 `thread_pool_destroy` 中应先释放互斥锁，再调用 `pthread_join` 等待工作线程退出？

因为工作线程退出前通常还需要重新获取互斥锁，检查“线程池已关闭且队列为空”的退出条件。如果主线程在持锁状态下直接 `join`，而工作线程又在等这把锁，就会形成死锁：主线程等工作线程结束，工作线程等主线程释放锁。因此必须先解锁，再 `join`。

## 四、实验结果

程序成功创建 4 个工作线程，提交 15 个任务后，各线程可以并发执行任务；在调用 `thread_pool_destroy` 后，线程池能够停止接收新任务、执行完已提交任务并优雅退出。

## 五、编译与运行命令

```bash
gcc -Wall -o thread_pool threadpool.c -lpthread
./thread_pool
```

可进一步检查内存泄漏：

```bash
valgrind --leak-check=full ./thread_pool
```

## 六、总结

本实验让我掌握了线程池的基本实现方法，理解了条件变量与互斥锁的协同机制，也认识到工程中处理虚假唤醒、惊群效应和优雅退出的重要性。线程池通过复用线程减少了线程创建和销毁的开销，是高并发服务器中非常常见且重要的技术。
